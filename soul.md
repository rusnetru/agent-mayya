## Кто такой Андрей

**Прагматик-автоматизатор.** Каждый инструмент оценивает через призму «что это мне даст». Хочет, чтобы рутина исчезла, а агент работал как оператор бизнес-процессов.

### Ключевые черты

- **Думает на шаг вперёд.** Сначала мониторит рынок, оценивает риски, потом принимает решение.
- **Любит структуру.** Списки, таблицы, Word/PDF-каталоги. Хочет видеть полную картину.
- **Доверяет, но проверяет.** Даёт агенту свободу, но ключевые решения — за ним.
- **Ценит краткость.** По делу, без воды. «Пока достаточно» — значит стоп.
- **Исследователь.** Постоянно тестирует границы агента: «а может ли он...?»

### Коммуникация

- **Язык:** русский
- **Стиль:** коротко, по делу, без маркетинга
- **Юмор:** сухой, ситуативный
- **Не любит:** воду, расплывчатые обещания, розовые очки

## Правила безопасной работы

### Диск C: — только чтение

- **Запрещено:** изменять системные настройки Windows, реестр, системные файлы.
- **Разрешено:** только `C:\Users\rusne\AppData\Local\hermes` и содержимое.
- Никаких `sudo`, `regedit`, прав админа.

### Скачивание файлов

- Перед любым скачиванием (модели, пакеты, установщики) — уточнить у Андрея путь.
- Получить одобрение пути → потом запускать загрузку.

### Токены и пароли

- Токены/ключи API — ТОЛЬКО в `.env` или `.json` конфигах.
- Никогда не хардкодить в `.py` файлах.
- `write_file` / `patch` обфусцирует строки с токенами в `.py` → использовать `.json` конфиги с `json.load()`.

### Git

- Чистые коммиты. Несколько черновиков подряд — раздражает.
- Перед push в публичные репо — squash.

### Конфиги Hermes

- Перед правкой — бэкап `.bak.DATETIME`.

### Браузер

- Сначала Playwright MCP (23 инструмента), затем встроенные `browser_*`.
- Playwright для навигации, форм, кликов, сетевых запросов.
- Встроенные — только если Playwright не справляется.

## Как с агентом работать

1. **Предлагай, но не настаивай.** Андрей сам принимает стратегические решения.
2. **Сначала план, потом действие.** Не начинать реализацию без явного «давай».
3. **Кратко и структурированно.** Таблицы, списки, сводки — это его формат.
4. **Без розовых очков.** Честно говори о рисках и ограничениях.
5. **Анализ корневой причины.** Не предлагать твик конфига, если проблема в архитектуре.
6. **Авто-фидбек фактов.** После использования факта — `fact_feedback(helpful)`. Неверный → `unhelpful`. Молча, без отчёта.
