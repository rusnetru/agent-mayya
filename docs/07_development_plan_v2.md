# План развития Mayya v2 — «от живого агента к передовому»

Дата: 2026-07-04 | Автор: Claude Fable 5
Статус кодовой базы на старте: 132/132 теста, живой диалоговый цикл, ~60 инструментов
(свои + MCP: браузер, GitHub), cron, под-агенты, навыки.

## Честная самооценка (зачем этот план)

Архитектурно Mayya — уровень современных практик. Реальные разрывы с передовыми агентами:

| # | Разрыв | Следствие |
|---|--------|-----------|
| 1 | Нет сжатия контекста | На длинной задаче забывает начало диалога |
| 2 | Память ищет по ключевым словам | «4-уровневая память» слабее, чем звучит |
| 3 | Нет стриминга | Пользователь смотрит на спиннер вместо ответа |
| 4 | Только терминал, cron живёт пока открыто окно | Автономность не доходит до пользователя |
| 5 | Нет evals | Прогресс качества не измеряется |
| 6 | Модель deepseek-chat | Потолок интеллекта (принимаем как данность бюджета) |

## Фазы

### Фаза 14 — Сжатие контекста (compaction)
Старая часть диалога не выбрасывается, а сворачивается LLM-конспектом
(факты, имена, решения, незакрытые задачи, ≤1500 символов), который живёт в system prompt.
Последние ~20 сообщений — дословно. Конспект накапливается инкрементально.
**Критерий:** диалог в 50+ сообщений не теряет имя пользователя и суть ранних договорённостей.

### Фаза 15 — Семантическая память (ChromaDB)
`ChromaVectorIndex` уже написан (all-MiniLM эмбеддинги) — подключить его к episodic-памяти
в проде (main.py), с мягким fallback на hash-индекс, если chromadb недоступен.
Персистентность: папка `chroma_db/` (gitignored).
**Критерий:** recall находит эпизод по смыслу («как зовут пользователя» ↔ «меня зовут Руслан»),
а не только по подстроке.

### Фаза 16 — Стриминг ответов
`complete_with_tools(stream=True)`: сборка text-дельт и tool_call-фрагментов из потока.
В терминале ответ печатается по мере генерации; вызовы инструментов показываются строками
статуса («🔍 web_search: …»). Событийный callback в ConversationalAgent.chat().
**Критерий:** первый токен ответа виден до завершения генерации; тесты сборки дельт зелёные.

### Фаза 17 — Telegram-режим
`python -m src.main --telegram`: long polling через Bot API (чистый urllib, без зависимостей).
Токен — свой бот Mayya (`TELEGRAM_BOT_TOKEN` в .env; у Hermes бот свой, токен не переиспользуем —
две long-polling сессии на одном токене конфликтуют). Фильтр по `TELEGRAM_ALLOWED_USERS`.
Cron-задачи в этом режиме шлют результат в чат — дайджесты приходят в телефон.
**Критерий:** диалог и cron работают в Telegram; без токена — понятная инструкция, не падение.
**Зависимость:** нужен токен от @BotFather (единственный шаг, который делает пользователь).

### Фаза 18 — Evals
`evals/`: ~8 сценариев (проактивность, поиск, память, инструменты, честность при неудаче)
с автоматическими проверками. Запуск вручную (`python -m evals.run`), результат — таблица
+ JSON с датой. Гоняем после заметных изменений, сравниваем с прошлым прогоном.
**Критерий:** первый замер сделан и закоммичен как базовая линия.

## Бэклог (после v2)
- Безопасность: подтверждение опасных команд (rm, форс-пуш), песочница для python_exec
- Постоянный сервис (автозапуск, Mayya живёт всегда — cron становится настоящим)
- Vision (скриншоты браузера → анализ)
- Голос (TTS/STT), веб-интерфейс
- Прокачка модели: маршрутизация сложных задач на более сильную модель
